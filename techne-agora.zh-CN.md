# Techne —— 每个项目是一件造物，每件造物是下一件的基石

[English](techne-agora.md) | [简体中文](techne-agora.zh-CN.md)

> 一篇立在广场上的文章：Techne 典则是什么，以及它为什么想走出 skill 的老路。

**这是篇观念文。** 你可以把它原样丢给你的 LLM Agent（Claude Code、Codex、Kimi……）去传话；完整的典则就在这个仓库里——[`techne-arche.md`](techne-arche.md)（太初）——你的 Agent 可以从头开始，跟你一起搭建造物。英文版：[techne-agora.md](techne-agora.md)。

## 核心想法

今天大多数 Agent 会话，结局都差不多：Agent 交给你一些文件。文件散落在某处；推理过程早已蒸发。让下一个会话——哪怕是*同一个* Agent——来接续、修改或审计，它只能干考古：哪个是最终版？参数怎么设的，为什么？试过什么、又失败过什么？什么都没留下；每个会话都要从零开始重建世界。

业界现在的答案是 **skill**：一份打包好的"怎么做"（一个 `SKILL.md` 加若干附件），教 Agent 一套流程——做演示、填 PDF、建表格。Skill 确实有用。但请注意它*是什么*：一页关于任务的指令。它对任务**留下什么**只字未提。Skill 告诉 Agent 怎么盖楼；它不定义什么是楼。一场用了 skill 的会话，产出仍然是一堆没有固定地址、没有出处、没有记忆的工件——下一个读者（人或机器）根本无从下手。

Techne 换了个角度。Agent 干活的单位不是任务，也不是 skill，而是**项目**——且每个项目是一件**造物**：

- **自包含**——被引用的一切都在造物之内；
- **可复现**——每个派生物都有一条可执行的链，回溯到不可变的源；
- **结构同构**——与其他每件造物结构完全相同：学会一个，即学会全部；
- **态迹两清**——当前态不含历史；历史在别处只增不删；
- **界面单点**——一个配置文件吸收全部机器／账号／平台差异；
- **初次到访即可读**——人或 AI：同一类读者，拒绝口口相传；
- **可复合**——造物复合为新造物，复合体仍是造物。

这不是七个特性，而是七条**公理**（A1–A7）；其余一切——十二定理、二十二条推论、典则十型、九条 SOP（SOP-0..5、SOP-7..9）——都是*推导*出来的，不是拍脑袋定的。任何无公理出处的规则都不成立。

## 为什么不是 skill

Skill 与造物回答的是不同的问题。这正是要害。

| | **Skill** | **Techne** |
|---|---|---|
| 问题 | Agent *怎么*做任务 X？ | X 留下的项目*是什么*？ |
| 本质 | 程序性说明书 | 存在形式的宪法 |
| 内容 | 指令（可选资源） | 源、推导、产物、文档、轨迹——整个项目 |
| 状态 | 无状态；每次调用冷启动 | 态与迹分离；记忆跨会话复利 |
| 可复现 | 尽力而为 | 定理（T3）：同输入同界面 ⇒ 同输出 |
| 可审计 | 无 | 三条独立路径：重算、回溯、读记录（T9） |
| 复合 | 无——skill 不可复合 | 造物可复合，复合体仍是造物（T10，满足结合律） |
| 接管 | 设计上不考虑 | 设计目标仅此一件：被未知的未来读者无损接管 |
| 信任模型 | 信任作者 | 运行检查（`verify_checksums`、`purity_scan`、SOP-9 清单） |
| 加载 | 渐进披露 | 分层阅读协议（同一思想的一般化） |

五个命题随之成立。

**1. 任何 skill 可无损内化为造物。** 指令成为方法学层唯一正文中 skill 锚点节；脚本成为 `script/utils/`；模板与参考文件成为 `data/ref/`；使用约定成为一条 SOP 或常驻 Prompt 的一节。Skill ⊂ Techne。

**2. 没有任何造物能由 skill 表达。** skill 格式无处安放：附校验和的不可变源（A1）、确定性推导链（C1）、态迹两清（A5）、父造物谱系（T10）、只增不删的决策记录（HANDOFF）。包含是真包含：**Skill ⊊ Techne**。

**3. 因此目标不是 skill，而是 skill 之后的形式缺失。** Techne 不与好流程竞争；它给每个流程的产出赋予*形式*——可寻址、可复现、可审计、可复合。若说 skill 是 Agent 学会一门手艺，造物就是这门手艺所在世界的物理学。

**4. Techne 无须额外维护——使用造物*就是*维护造物。** skill 是身外之物：与它所描述的工作分开书写，于是与现实脱钩、腐烂。造物则由工作*本身*维护——在典则之下进行的每一次构建、分析、修复与完善，都让这件造物更完整：HANDOFF 更丰厚、methodology 更锐利、产出更可复用。没有维护预算可言，因为项目的维护*就是*造物的维护。且因造物可复合（T10），昨日维护良好的造物成为今日不可变的源：每个项目都为将来接续它的每个项目增添永久的、可复用的价值。skill 的维护是 recurring cost（反复发生的成本）；造物的维护是 compound interest（复利）。

**5. skill 随模型贬值，造物不会。** skill 编码的是一套按"今日模型尚不能为"校准的流程；模型一进步，绝大多数 skill 悄然过期——流程变成原生能力，说明书成了博物馆藏品。造物不编码任何流程：它规定的是结果必须*成立*的东西——形式、出处、可审计性——无论 Agent 能力几何。skill 回答的问题（教模型做任务）是模型相关的，随模型变强而缩小；造物回答的问题（被未知的未来读者跨机器、跨时间无损接管）是模型无关的，不随模型变强而缩小。更强的模型让宪法*更*必要，而非更不必要：Agent 越强大，约束其产出的形式越有价值。典则自身也可修正（它自带修正程序），但此处的修正是生长，不是腐烂。skill 如教程般老去；造物如法律般长存。

（Karpathy 的 `llm-wiki` 模式从另一领域独立地抵达了同一论点——见《为何有效》。）

## 架构

本体系立于三块基石，各有一个希腊名与一份专职：

| 文件 | 角色 | 词源 |
|---|---|---|
| [`techne-arche.md`](techne-arche.md) | 太初——典则，完整的公理体系 | ἀρχή：第一原理、本原、治权 |
| [`techne-daemon.md`](techne-daemon.md) | 守护灵——常驻 Prompt，每轮完整读取 | δαίμων：守护灵；常驻进程 |
| [`HANDOFF.md`](HANDOFF.md) | 迹——状态快照＋只增不删的倒序日志 | — |

典则自身是一座推导塔：

```
A1 源始 origin       A2 构造 construction   A3 确定 determinism   A4 类型 type
A5 态迹 state/trace   A6 界面 interface      A7 明读 legibility
        │
        ▼  十二定理（T1–T12）
  T3 可复现 · T6 结构同构 · T9 审计三径 · T10 造物复合 …
        │
        ▼  二十二条推论（C1–C22）
  C1 验证链 · C5 态纯 · C12 数据集等价 …
        │
        ▼  构造典则——十型，一切项目同一棵树

<项目>/
├── README.md      使用知识（态）            ├── config/    唯一机器特定点
├── HANDOFF.md     决策轨迹（只增不删）        ├── script/    推导：pipeline · datasets · utils
├── source/        给定的、不可变的、附校验和   ├── document/  research · reports · methodology ·
├── data/          ref · pipeline · unformatted │             metadata · dataset · parameters
├── output/        命题断言型产物              ├── panel/     原始可视化
└── figure/        仅允许声明的手工润饰
```

每个项目——组学分析、文献综述、软件构建、知识库——都是同一棵树的实例。阶段槽位允许空置；类型永不按项目合并、改名或私创。

## 操作

**四步闭环**（A7·SOP-0·T9·C17·C22）——任何 techne 项目的完整工作流：

1. **触发**（首次）：`/techne-arche` → Agent 读典则，建立体系；然后描述你的项目；
2. **循环**（每轮）：`/techne-daemon` → Agent 读守护灵，按典则作业；继续用自然语言提示词对话；
3. **审计**（收口，形式）：`/techne-dokimasia` → 八镜片并行审验（t7/c11/a7/c8/t9/rg/c20/c21），C20 形式挂靠；零 violation 方可交付；若不满足则修复后重跑第 3 步；交付仍未闭合则自第 2 步再循环；
4. **审查**（收口，事实）：`/techne-epikrisis` → r20 单 sub-agent（**不与③并行**避免单轮 LLM 注意力稀释）；C22 完整性与事实有效性双校验；零 violation 方可交付。

闭环之内，日常作业机制不变：

**启动（SOP-0）**：新会话先读 README → 读 HANDOFF 顶部 → 读 config → 输出对齐结论（什么可复用、什么需新建、什么需调整）。未对齐，不动工。

**委托**：项目复合由 T10 给出（A1–A7·T5 直接推论）。构建复合项目时，本项目 source 可接纳上游项目的果（附校验和与谱系声明）；禁止修改上游以适配下游。一切适配发生在下游的界面层与特异应用层。

**作业（SOP-2）**：每个决策当日入 HANDOFF，附理由与候选。每个工件立即落入其型之位。外部依赖：内化入 `data/ref/` *或*登记于 config——二选一，无第三条路。非确定输入（API 响应、人工判读、抽样）先冻结为附校验和的源／数工件，*再*入链。

**收尾（SOP-3）**：重跑触及的链 → 完成当日 HANDOFF 条目 → 态内容清除过程痕迹 → 一次完整意图一次原子提交 → 对照 SOP-9 验收清单。交付物自身必须是一件可复合的合规造物。

**审计（SOP-7）**：三条独立路径交叉验证：重算比对（正向）；沿链回溯至源并核对校验和（逆向）；读 HANDOFF 中"为何"的叙事（叙事径）。审计门属用户主权：主 agent 提醒，用户调 `/techne-dokimasia`。

## 阅读协议（守护灵）

每个会话通读典则全文*并非*最佳实践——典则自己这么说。首次接触：完整读一遍 [`techne-arche.md`](techne-arche.md)（太初），建立体系。常驻：[`techne-daemon.md`](techne-daemon.md)（守护灵）——约 70 行的操作核投影，头部四条款声明它是*投影*、永非第二真相源——每轮读取。按任务取相应 SOP。全文复读仅限三种情形：首次接触、典则版本变更、对典则做元工作。

## 一个实例：本仓库

Techne 以自身的元项目发布——**techne-demiurge**（δημιουργός，依照永恒形式营造世界的工匠）。你正在读的这个仓库，就是一件生产典则的造物。它的 `source/` 是空实例——态迹两清推至极限的形态：历史记录只增不删地存放于 git 历史与 [`HANDOFF.md`](HANDOFF.md)，公开可审计；典则在原则上以自身为起点。`document/research/` 存有本项目不可还原的调研文档（C15）；`document/reports/` 存有典则迭代的结果报告；`HANDOFF.md` 记录每个决策及其理由——本仓库每一项主张背后的公开审计轨迹。这条纪律凝成一句无时效的规则：*不凭记忆推送，一切以哈希验证。*

## 技巧与提示

- **验证源**：`bash script/utils/verify_checksums.sh`——让"不可变"可被检验。
- **扫描态纯**：`bash script/utils/purity_scan.sh`——在典则中检索修订痕迹、版本词与不得存于态中的外部指涉。
- **串行编辑**：对同一文件的并行编辑会竞态；串行编辑可避免并发写丢失。串行。
- **不凭记忆推送**：任何传输前当次读盘；只接受"blob SHA 一致"为唯一验收标准。
- **命名**：家族成员取希腊／拉丁词根——ἀρχή 太初（典则）、δαίμων 守护灵（守护）、δημιουργός（营造者）、ἀγορά（本广场）。词源考证存于 HANDOFF。

## 为何有效

项目工作中枯燥的部分不是思考——而是簿记：命名、归档、记录决策、让当下免于历史、让依赖皆有声明。人类放弃这种纪律，因为维护成本涨得比价值快。Agent 不会厌倦——但它*会*遗忘，因为它没有过去。Techne 的答案是：让簿记成为**结构**（靠类型，不靠意愿），让记忆成为**一个文件**（HANDOFF，而非聊天记录），让规则**可推导**（七条公理，而非七百条惯例）。Agent 永远不必记住"必须记录"；它已读过的 SOP 告诉它*何时*记，形式告诉它*记在哪*。

值得一提的是 Karpathy 的 [`llm-wiki`](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) 模式在个人知识领域独立地重新发现了同一组范畴：不可变的 *raw sources* ≙ `source/`（A1）；持续复利的 *wiki* ≙ 项目之态（A5）；*schema*（`CLAUDE.md`）≙ `techne-arche.md`＋`techne-daemon.md`；面向内容的 *index.md* ≙ `README.md`；编年只增的 *log.md* ≙ `HANDOFF.md`；*Lint* 操作 ≙ SOP-7＋`purity_scan`；"好答案回填 wiki" ≙ 结果落 `document/reports/`（T12）。`llm-wiki` 是造物在知识库域的实例化；Techne 是该模式的公理化一般形式——适用于 Agent 交付的一切领域。

## 适用范围与边界

诚实的一节。Techne 并非零成本：典则在首次接触时消耗上下文（此后守护灵很便宜）；对没有交付物的一次性问答，完整 SOP 纪律是杀鸡用牛刀。对*无状态微流程*——如何调一个 API、如何格式化日期——skill 仍是更轻的工具，因为没有什么项目可留下。二者可以很好地复合：保留你的 skill，把它们归档进造物（`document/methodology/`），它们便获得出处、审计与记忆。这里的主张不是"删掉你的 skill"；而是：skill 范式若单独存在，Agent 的工作将永远无形——而形式，是可以推导出来的。

本文有意站在典则之上一层。精确的结构、十型、SOP、验收清单——全部自含于 [`techne-arche.md`](techne-arche.md)（太初）；操作最小集在 [`techne-daemon.md`](techne-daemon.md)（守护灵）。一切模块化：为你的领域实例化这棵树，留取有用之物，让 HANDOFF 记录你改了什么、为什么。本文唯一的职责是传播这个模式——若说本文是预告片，太初即宪法全文、守护灵即贴在工位上的每日检查卡——并在公开的广场上宣告：无形交付物的时代可以终结了。
