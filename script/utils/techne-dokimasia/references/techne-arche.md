# Agent 项目架构设计哲学

## ——造物典则（Canon of Technes）· 太初

> **地位**：本典则是一切由 Agent 构建与交付之项目的统一架构宪法——不限领域、不限技术栈、不限规模。任何 Agent，在任何设备、任何平台的任何新会话中开始任何任务之前，先阅读本典则；一切项目按本典则规定的标准接口交付；交付结果本身仍是一件可继续复合的造物。
>
> **体系信条**：用户可以用一件件造物搭建一个个新的项目，新的项目本身也是一件造物。此非愿望，而是本典则的定理（T10）。

## 第〇层　操作核（Bootstrap）

**追溯说明**：本层是全文的操作性摘要与索引，不引入任何下文未证的概念；每一条均可沿标注回溯至公理—定理—推论链。

### 0.1 一句话

**每个项目是一件造物：源不可变、链可复现、结构同构、态迹两清、界面单点、人与 Agent 同等可读、可与其他造物复合为新的造物；全部规则由七条公理推导生成，任何无公理出处的规定都不成立。**

### 0.2 每次会话的启动协议（详见 SOP-0）

1. 读本典则第〇层与第五层——获得**形式**；
2. 读目标项目 `README.md`——获得**上下文、用法与谱系**；
3. 读目标项目 `HANDOFF.md` 顶部——获得**现状与最近决策**；
4. 读目标项目 `config/` 环境层——获得**环境**；
5. 输出对齐结论（哪些造物可复用、什么需新建、什么需调整），然后开工。作业遵 SOP-2，收尾遵 SOP-3，交付对照 SOP-9 验收。

### 0.3 体系一页速览

## 第一层　元层：目的、第一性原理与概念原语

**追溯说明**：本层定义全部概念原语，回答"为何需要本典则"；不引用任何后续层的结论。

### 1.1 目的

本典则解决唯一的问题：

> **如何让任意项目，被任意未来主体（人或 Agent），在任意设备、任意平台、任意新会话中，无损地接管，并作为造物继续复合。**

"无损接管"分解为四个可判定的子目标。此处仅宣告目标；其可实现性由公理系统**证明**，而非规定：

1. **物质保真**——项目的全部产物可被重建（→ 复现定理 T3）；
2. **认知保真**——第一次到来的读者可完整获得项目的全部知识（→ 明读公理 A7、分工定理 T12）；
3. **形式经济**——接管的认知成本与项目数量无关（→ 同构定理 T6）；
4. **功能复用**——项目可嵌入未来的项目（→ 积木定理 T10）。

### 1.2 自省：工程实践中的六条行为逻辑

以下六条行为逻辑在大量项目实践中反复有效，本典则将其抽象为公理：

1. **以"物"统"事"**。按工件类型而非流程步骤组织项目。行为逻辑：流程易变，类型稳定；只有稳定的形式才能承载多变的内容。→ 抽象为类型公理 A4。
2. **把"时间"放逐出当前态**。行为逻辑：读者——尤其是新会话中的 Agent——**没有过去**；项目必须在读者没有过去的情况下依然完整、连贯、当前。→ 抽象为态迹公理 A5。
3. **把"机器差异"压缩为单点**。行为逻辑：变化不可避免，但变化的*位置*可以选择；变更局部化是可维护性的全部秘密。→ 抽象为界面公理 A6 与单点定理 T8。
4. **把"复现"定义为构造而非承诺**。行为逻辑：不能被重建的结果，在认识论上等于不存在。→ 抽象为构造公理 A2 与复现定理 T3。
5. **多对象等价对待**。行为逻辑：特例是结构腐烂的起点；等价性——多个同类对象被完全相同地对待——是消除特例的唯一手段。→ 抽象为同构定理 T6 与推论 C12。
6. **知识与历史分居**。行为逻辑："知识"与"历史"是两种不同的认识对象：知识经压缩后仍为真；历史不可压缩——决策的理由、排查的轨迹、参数选择的依据，一旦压缩即失真。→ 抽象为分工定理 T12。

### 1.3 第一性原理

> **项目 ＝ 信息 ＋ 推导 的系统；其全部价值在于被未知的未来主体无损接管与复合。因此，一切架构规则都是同一个问题的解：跨主体、跨时间、跨机器的知识传递问题。**

此传递问题的不可约维度恰为六个——**存在**（何物存在）、**因果**（由何而来）、**形式**（取何形态）、**时间**（何时而变）、**空间**（居于何处、内外之界）、**认知**（如何被知）。第二层的七条公理与之一一对应（"存在"需两条：根基 A1 与形式 A2）。此为公理系统维度完备性的直觉论证；操作性的完备性论证见 §2.9 及附录 A。

### 1.4 概念原语（定义 D1–D12）

- **D1 工件（artifact）**：项目内任一具名、具址、具型的内容单元（文件为典型载体）。
- **D2 推导（derivation）**：可执行的变换；以若干已存在的工件为输入，产出工件。其物质载体为 `script`。
- **D3 类型（type）**：工件的范畴；决定其语义、规范位置与可参与的推导。
- **D4 环境（environment）**：项目边界之外的一切：机器、操作系统、解释器、账号、平台、网络、其他项目、会话记忆。其本质特征是**异质性**。
- **D5 界面（interface）**：项目与环境之间唯一被显式声明的交换点；其物质载体为 `config`。
- **D6 读者（reader）**：具备通用能力、对项目无既有记忆的主体——人或 Agent。按行为分三角：**构建者**（产出制品）、**审计者**（不修改地验证合规）、**使用者**（消费已交付制品）；同一实体可兼具多角；人与 Agent 的权利与义务完全相同。
- **D7 态（state）**：项目的当前状态工件集合。
- **D8 迹（trace）**：项目变更的记录集合（HANDOFF 与版本历史）。
- **D9 项目（project）**：满足公理 A1–A7 的（工件，推导，源，界面）四元组。
- **D10 积木（techne）**：可被其他项目复合、且复合产物仍为项目的项目（其存在性由 T10 保证）。
- **D11 冻结（freeze）**：将非确定输入固化为附校验和的 source／data 工件，再进入推导链。
- **D12 谱系（genealogy）**：项目之间（父积木）与项目之内（构造坐标）的时间坐标系。

### 1.5 记法

- `X ⊢_d a`：推导 d 以工件集 X 为输入，产出 a；
- `src`：源集；`τ(a)`：工件 a 之型；`loc(τ)`：型 τ 的规范位置；
- `Q∘P`：项目复合（Q 之源接续 P 之果）；`⇒`：推出；`∅`：空；
- 推导链标注 `［A2·A3 → T3 → C1 → §5.4］` 读作：公理 A2、A3 证出定理 T3，T3 生出推论 C1，落实于 §5.4。

## 第二层　公理层

**追溯说明**：本层仅使用第一层原语。七条公理互相独立（§2.8），联合覆盖六个不可约维度（§2.9）。它们在逻辑上不可还原——删除任何一条，必有目的层（§1.1）宣告的性质随之崩塌；任何其他表述的"公理"均可被证明为本系统的定理（附录 A）。

### 2.1 公理 A1（源始）——存在的根基

- **形式**：每个项目含非空源集 `src`；`src` 中每一工件满足：（i）在项目内不可推导（`¬∃(X, d)：X ⊢_d s`）；（ii）不可变更（写入之后，任何修改即构成一个新项目）。
- **白话**：一切必有给定的起点；起点不能被项目自身制造，也不能被事后改写。
- **不可还原**：缺之，则推导无根，"复现"失去比较基准——T3（复现）、T9（审计）崩塌。
- **排除**：凭空产物；改写历史输入；以派生物冒充原始物。
- **直觉**：时间单向；计算的锚点必须先于计算存在。

### 2.2 公理 A2（构造）——存在的形式

- **形式**：每个非源工件 a，皆是且仅是项目内推导的输出——存在从 `src` 出发抵达 a 的推导链；且推导是构造性的：推导之输入必先于其输出存在（构造关系无循环）。
- **白话**：项目里不允许"魔术文件"：每个派生物都有构造它的程序；不允许循环依赖。
- **不可还原**：缺之，则允许无构造者的工件存在——T2（分层）、T9（审计）崩塌。
- **排除**：来路不明的产物；未声明的手工修改（唯一允许的声明式例外见 C2）。

### 2.3 公理 A3（确定）——因果的秩序

- **形式**：推导是确定性映射：同一推导、同一输入、同一界面参数 `⇒` 同一输出。
- **白话**：重跑必得同果。一切非确定输入（外部 API 响应、人工判读、随机抽样、会话中生成的中间判断）必须先**冻结**为源／数工件并附校验和，再进入推导链（C7）——把不确定性挡在项目的边界上。
- **不可还原**：缺之，则"重执行"失去意义——T3（复现）、T4（幂等）崩塌。
- **排除**：隐藏随机性；未冻结的外部调用；依赖执行时刻的状态。

### 2.4 公理 A4（类型）——形式的秩序

- **形式**：存在有限类型集 𝕋；每一工件恰属于一个型；型决定工件的语义、规范位置 `loc(τ)` 与可参与的推导。
- **白话**：万物各归其位；位置即语义。由此，每一工件获得环境无关的稳定标识（类型定址），同一概念在项目内任何位置使用同一名称。
- **不可还原**：缺之，则"位置"与"等价"无定义——T6（同构）、T8（单点变更）崩塌。
- **排除**：一物多型；无型之物；项目私创顶层新类型（类型演替是典则级事件，见附录 C）。

### 2.5 公理 A5（态迹）——时间的秩序

- **形式**：项目工件分为态与迹，二者不交；态中不含迹的内容（无修订记录、版本标注、会话痕迹）；迹不改变态；迹只增不删（append-only）。
- **白话**：任何时刻进入项目的读者，看到的都是连贯的"现在"；全部"过去"被完整隔离在指定居所（HANDOFF 与版本历史）。
- **不可还原**：缺之，则当前态被历史污染，读者被迫考古——T7（两清）、T12（分工）与可维护性崩塌。
- **排除**："更新版／v2／已纠正"字样；过程性批注混入态；删改历史记录。

### 2.6 公理 A6（界面）——空间的秩序

- **形式**：项目与环境的全部交换，经由唯一显式声明的界面发生；界面之外，项目内任何工件不依赖环境之具体形态（机器、路径、账号、平台、他项目、会话记忆）。
- **白话**：对内自包含，对外单点接触；换机器、换账号、换平台，只改一个文件。
- **不可还原**：缺之，则环境耦合散布全项目——T5（闭包）、T8（单点变更）、SOP-8（迁移）崩塌。
- **排除**：硬编码绝对路径；指向他项目的引用；第二个"看似机器配置"的文件。

### 2.7 公理 A7（明读）——认知的秩序

- **形式**：项目自含充分文档，使读者（D6）无需项目外信息即可完成四件事：**理解**（目的与原理）、**使用**（每个可执行内容）、**复现**（逐步重建产物）、**扩展**（在其上继续工作）。
- **白话**：项目必须为一个"没有过去、没有记忆、第一次到来"的读者而写；人与 Agent 是同一类读者。
- **不可还原**：缺之，则形式完备而认知封闭——SOP-0（启动）与积木接管（T10）崩塌。
- **排除**：隐含上下文；口口相传的知识；仅存于会话记忆中的决策。

### 2.8 不可还原性总证

方法：对每条公理 Ai，构造一个满足其余六条而违反 Ai 的模型，观察何种性质丧失。

| 删去 | 反模型 | 随之崩塌 |
|---|---|---|
| A1 | 全部工件互相推导（无根之环） | 复现无锚：T3、T9 |
| A2 | 允许无构造者的工件 | 来源不明：T2、T9 |
| A3 | 推导含隐藏随机性 | 重执行失真：T3、T4 |
| A4 | 工件无型或一物多型 | 位置与等价无定义：T6、T8 |
| A5 | 态内嵌历史 | 读者被迫考古：T7、T12 |
| A6 | 环境耦合散布全项目 | 迁移即重写：T5、T8 |
| A7 | 知识依赖口头与记忆 | 新主体无法接管：SOP-0、T10 |

七条公理互相独立，任何一条都不是其余各条的推论。∎

### 2.9 完备性（操作定义与论证）

本系统不声称逻辑学意义上的强完备，而声称**生成完备**，其操作定义为两条可检验的断言：

1. **覆盖性**——本典则的每一条规则均可沿标注的推导链回溯至公理，无悬空；工程实践中提出的一切候选原则与候选"公理"均可还原至本系统（附录 A），无剩余；
2. **可追溯性**——自第三层起，每一条规则都标注其推导链，任何规则可在有限步内回溯至公理。

维度论证：任何关于项目的规定必回答六问之一——何物存在、由何而来、取何形式、何时而变、居于何处、如何被知；七公理与六问的对应为 A1/A2→存在、A3→因果、A4→形式、A5→时间、A6→空间、A7→认知。故不存在游离于公理维度之外的规则类型。∎

## 第三层　定理层

**追溯说明**：每条定理以「陈述／推导／白话」三段给出；推导仅引用公理与在先定理。定理是公理的必然展开，不含任何新的约定。

### T1　根基定理（唯一再生基）

- **陈述**：`src` 是项目的唯一最小再生基——仅由 `src`、`script` 与 `config`，可原则性重建项目的全部态；删去 `src` 中任一工件，必有派生物不可复现。
- **推导**：A2 给出从 `src` 到每个派生物的链；A1 保证 `src` 不被重建亦不被改写，故链的端点唯一。若某 `src` 工件不被任何推引导用，则它不属于再生基，应退出项目；任一被引用的 `src` 工件被删，以其为输入的推导即失其输入。［A1·A2］
- **白话**：source 是项目的"公理集"，其余一切是它的"定理集"。

### T2　分层定理（推导图是 DAG）

- **陈述**：推导关系构成以 `src` 为根层的有向无环图；全部工件可按构造先后拓扑分层：L₀＝`src`，L₁，L₂，…。
- **推导**：A2 规定输入先于输出存在，构造关系因而良基（无循环、无无穷下降）；有限项目上的良基关系必可拓扑排序。［A2］
- **白话**：项目有一条从源到果的"时间之河"，每件产物都站在确定的河段上；依赖方向永不倒流（推论 C13；命名制度见 C10）。

### T3　复现定理

- **陈述**：对每一派生物 a，存在构造性证明：一条从 `src` 到 a 的推导链；在同一界面参数下重执行该链，再生与 a 等价的产物。
- **推导**：T2 给出链的存在；A3 保证重执行结果相同；A6 保证链不依赖界面之外的环境，故界面参数固定后，链可跨机器重放。［A2·A3·A6·T2］
- **白话**：复现不是美德，是公理的推论；不能复现的产物等于不存在。

### T4　幂等定理

- **陈述**：重复执行任一推导链是安全的：产物在其规范位置原位再生，不扰动其他工件。
- **推导**：A3 给出输出同一；A4 给出输出位置唯一（`loc(τ)`），故重写即覆盖自身；A6 排除界面外的副作用。［A3·A4·A6］
- **白话**：脚本可从任意中间点重跑；跑两遍等于跑一遍（执行协议见 C14）。

### T5　闭包定理（自包含的精确形式）

- **陈述**：项目对推导关系封闭：所有推导的全部输入，或在项目之内，或经界面显式声明；不存在指向项目外的隐式引用。
- **推导**：A2 要求每个输入存在；A6 要求环境交换仅经界面。二者合并：凡被引用者，必被**内化**（复制为项目工件，入 `data/ref/`，附校验和）或**显式化**（登记为界面环境层参数，只读引用）。无第三条路。［A2·A6］
- **白话**：自包含 ≠ 与世隔绝，而是"一切外部依赖都被点名"。

### T6　同构定理（多项目结构完全等价）

- **陈述**：遵循同一典则（同一 𝕋 与 `loc`）的任意两项目，其目录树结构同构：存在保型、保址的节点双射；项目之差异仅在于内容，不在于形式。
- **推导**：A4 使每工件之型唯一、型之位置唯一；两项目共享 𝕋 与 `loc`，故其树形逐节点对应。［A4］
- **白话**：学会一个项目的结构，即学会所有项目的结构。"完全等价"是结构同构，不是内容同一；其价值在于把接管成本从 O（项目数） 降为 O(1)。

### T7　两清定理

- **陈述**：态无史，迹不动态——读者阅读态内容无需任何历史知识；审计者阅读迹内容无需重构任何历史现场。
- **推导**：A5 直接给出分离；迹只增不删，故迹自身连贯可读。［A5］
- **白话**：当前态永远"第一次阅读友好"；历史永远"审计友好"。

### T8　单点定理（变更局部化）

- **陈述**：（i）环境因素的任何变更，恰好需要编辑一个工件（界面环境层）；（ii）分析变体的增删，恰好需要增删一个工件（界面变体层），共享工件零编辑。
- **推导**：A6 将全部环境依赖集中于唯一界面；界面的双层构造（环境层／变体层，§5.4）使机器信息与方案信息正交，故各层变更互不及邻。［A6·A4］
- **白话**：可维护性的全部秘密，是变化的位置被预先选定。

### T9　审计定理（三径可溯）

- **陈述**：每个派生物的来源可完全重建，且有三条相互独立的路径：**正向**——重算并比对（T3）；**逆向**——沿推导链溯至 `src`（T2）；**叙事**——查阅迹记录中的决策理由（A5）。任何状态、任何制品、任何决策，均存在可执行方法在有限时间内判定其合规性。
- **推导**：T3 给正向，T2 给逆向，A5 保证迹完整且未污染态；三径独立，故可交叉验证；三径皆通即合规可判。［T2·T3·A5］
- **白话**：审计 ＝ 重算一遍 ＋ 走回去一遍 ＋ 读一遍"当时为什么这么决定"。

### T10　积木定理（复合封闭性）★

- **陈述**：设项目 P、Q 遵循同一典则。若 Q 的源接口接纳 P 的果（`output`），则复合 Q∘P——将 P 之果**复制**为 Q 之源（附校验和与谱系声明）——仍满足 A1–A7，即仍是积木；复合满足结合律；故"积木复合积木得积木"，可无限递归。
- **推导**：逐条验证七公理：A1——P 之果进入 Q 后是给定且不可变的源 ✓；A2——Q 的内部推导链不变，新增源依定义无需构造者 ✓；A3——复制与校验和冻结了上游结果，确定性保持 ✓；A4——共享典则保证类型一致 ✓；A5——复合这一行为记入 Q 之迹（HANDOFF 与谱系声明），态不受影响 ✓；A6——复合是"复制"而非"引用"，不产生跨项目路径，闭包由 T5 保持 ✓；A7——接缝（谁之果、何版本、何时接入）记入 Q 的 README 谱系节 ✓。结合律：(C∘B)∘A 与 C∘(B∘A) 在相同规范位置放置相同工件，树同即等。递归：复合产物满足全部公理，故仍是积木。［A1–A7·T5］
- **白话**：**积木性不是公理，而是定理**——它是自包含与类型化的涌现性质。"用户可以用一块块积木搭建新项目，新项目本身也是一块积木"，由此被证明，而非被规定。

### T11　正交定理（变体隔离）

- **陈述**：分析变体之间相互正交：任一变体的增删或重跑，不改变其他变体的任何工件。
- **推导**：变体层仅覆盖差异参数（T8-ii）；变体输出按 `<variant>/` 子目录隔离于各阶段型之下（A4）；共享文件零编辑（T8-ii）。［T8·A4］
- **白话**：新增一个分析 ＝ 新增一个文件，不动任何旧文件。

### T12　分工定理（知识单点）

- **陈述**：每一项项目知识恰有一个文档归宿：使用性知识 → `README.md`；轨迹性知识 → `HANDOFF.md`；方法性知识 → `document/`（methodology · parameters · research · metadata · dataset）；结果性知识 → `document/`（reports）。
- **推导**：A5 把知识分为态知识与迹知识；A7 要求态知识对读者充分且可得——理解一个项目，方法、结果与输入缺一不可（结果性知识 ＝ 产物之意义的综合与解读；数据集登记 ＝ 源之身份）；A4 要求各归其位。若同一知识有两处居所，漂移必至（两处终将不一致），与 A3 的同一性相悖，故居所唯一。［A5·A7·A4］
- **白话**：同一句话不写两遍；要引用，就链接。此即"单一真相源"与"最小冗余"的生成形式。

## 第四层　推论层

**追溯说明**：推论是定理的制度化表达——每条推论是一条可直接执行的规则，末注其定理出处。

- **C1 复现链**［T2·T3］：恒成立两条验证路径，每个环节由 script 衔接——
  `source →(script)→ data →(script)→ output →(script)→ panel →(script／声明的编辑)→ figure`
  `source →(script)→ data →(script)→ panel →(script／声明的编辑)→ figure`
  即：每个 panel 可由 data（＋output）＋script 复现；每个 output（发表级支撑数据的来源）由 data＋script 生成；data 由 source＋script 生成；document 记录每步方法学与参数取值的理由。链上任一断点皆可被定位与修复。
- **C2 声明的手工步骤**［T3·T9］：唯一允许的非可执行推导步骤是 figure 的润饰；它必须（i）最小化，（ii）在 methodology 中声明，（iii）在 HANDOFF 中记录。除此之外，任何手工改动下游产物皆为违约。
- **C3 闭包执行**［T5］：禁止指向项目外的引用（绝对路径指向他项目文件、跨项目记忆）；外部参考件（注释、映射、参考数据、外部资产）二选一——内化入 `data/ref/` 自包含，或登记于环境层经界面参数只读引用。选择标准：复现必需且体量允许者内化；体量过大或远端动态者显式声明，并将其所需切片冻结（C7）。
- **C4 同构执行**［T6］：新项目 ＝ 典则树的实例化；不增删、不改名任何顶层类型。
- **C5 态纯执行**［T7］：README／document／script／data／output／panel 等态内容永远呈现为当前态；禁含修订记录、版本标注（"更新版／已纠正／v2"）与会话过程痕迹；轨迹统一归 HANDOFF 与 git 历史。唯一例外：script 注释中允许"why"型解释（解释设计意图，非修订历史）。
- **C6 界面双层**［T8·T11］：`config` ＝ 环境层（`run_config`，唯一机器特定点）＋ 变体层（`<variant>`，每方案一件 run-spec）；新增分析 ＝ 新增一件变体文件，不改共享文件。
- **C7 冻结外部不确定性**［A3·T3］：一切非确定输入（外部 API 响应、人工判读、随机抽样、会话中生成的中间判断）先固化为 source／data 工件并附校验和，再入推导链。
- **C8 审计日常化**［T9·A1］：决策、参数选择、排查、迭代当日记入 HANDOFF；git 原子提交；source 附校验和文件并定时备份——不可变性必须可验证；每个运行步骤产生结构化日志（§5.4-10）。条目级 6 件齐备与秉笔直书义务见 **C21**（本条不复述）。
- **C9 变体隔离**［T11］：变体输出一律按 `<variant>/` 子目录隔离于 output｜figure｜panel｜data/pipeline｜document/metadata 之下，自包含、互不串扰；变体间仅共享环境层，禁止跨变体引用。
- **C10 链式命名**［T2］：链式中间数据按 `{ID}_{阶段}` 命名，阶段序即拓扑序；数据集 ID 稳定唯一。
- **C11 知识单点**［T12］：每项知识恰有一个居所；禁止重复文档；引用用链接，不复制。参数推荐值只出现于 `document/parameters/`，方法学解释只出现于 `document/methodology/`；脚本不复述参数范围，只经配置层读取。一名一实的“实”是**锚点**而非文件数；末端唯一正文内以二级标题锚定各脚本／组件（§5.3-4）。
- **C12 数据集等价**［T6·A4］：多数据源（多数据集）时，各类型目录的内容先按数据源分子目录（`<类>/<dataset>/`），再在该数据集内组织；统一生效于 source、panel、figure、output、data/pipeline、data/unformatted、document/metadata、script/datasets——多个数据集被完全相同地对待。唯一例外：跨数据集汇总面板入 `panel/_summary/`。
- **C13 依赖单向**［T2·A2］：依赖方向沿推导 DAG 永不倒流——`utils/` 不依赖 `pipeline/`，`pipeline/` 不依赖 `datasets/`；`datasets/` 是 `pipeline/` 的应用而非反过来；高层不感知低层的具体实现。
- **C14 幂等入口**［T4·T3］：每个可执行单元具有统一入口契约（§5.4-9）：失败即停；已存在的合规产物默认跳过，强制重跑由显式开关触发；脚本可从任意中间点重跑。
- **C15 调研不可还原**［T12·A7·T5·T6］：`document/research/` 下每一份文档必须形式完备（命题—论证—结论完整链）、逻辑自洽、不可由项目内其他文档还原（§5.4-7）；非结构化原始件、半成品笔记、单一方法复述不得入内。**闭包义务**：本项目论证**鼓励**挂可审计文献标识（DOI／PMID／accession 等，与 methodology 参考文献节一致，C20·禁令 9）。禁止的是**他项目内容泄漏**——把**其他项目**的实例表、实例值、未内化路径、未声明的外部项目／产品／skill／agent 专名当作本元项目态正文的证据堆砌；判例可用抽象字段与缺陷类别，或只用**本项目**已登记／已内化的标识。实例细节属于其实例项目或 HANDOFF，不回流他仓之元项目态（T5 闭包·T6 同构）。
- **C16 入口义务**［A7·T12］：每一知识居所设唯一入口，使读者无需先验即知从何读起——项目 → README 入口层；方法学层 → 该层唯一正文（`document/methodology/methodology.md`，或与之等价的层内唯一 md）；轨迹 → HANDOFF 状态快照。入口三要件：①读者路由（谁该读什么、按何顺序）；②整体叙事（本层知识如何拼成一个整体）；③域内索引／节锚点（指向而不复制，C11）。入口可以是层内唯一文件的文首，不必另立第二文件。

- **C19 根本性重构义务**［A2·A3·A5·T12·附录 A］：对任何合规造物的任何修复或改进，须遵循「内省→提炼理念→拆解→还原→推广→内化→重构」七步——从公理化体系源头审视缺口，能由既有公理／定理／推论推出则不新增条文；必须新增时按附录 A 还原后并入应在层级，改写宿主条款为单一连贯表述，删除被取代的旧句、例外编号与并行解释。禁止局部补丁（patch-on-patch）；迹保留完整决策史，态只留重构后的现在时。此义务适用于 Techne-demiurge 造物本身及一切基于运行时四角工作流的项目。**附录 C 的修正程序与本条同旨：一次推导输出再生后的当前态全文，不得「旧句＋补丁句」并存。**

- **C22 事实性内容审查义务**［A2·A3·A7·T9·T12·T3·§5.4-13·C20］：报告中**每一条已挂靠的证据（i / ii）**必须在事实层上能证明其紧前判断／结论；不得存于「仅路径存在而内容未核」或「证据支持有限而被断言为强」的状态。本条与 C20 互为补集——C20 审**形式挂靠**（路径／DOI 是否在场），C22 审**事实挂靠**（证据是否真支撑）。义务两维：(1) **完整性**——可证明该结论的项目内工件与文献须尽列，不可单边挑证；opposing 侧不得空置为默认赞同；(2) **事实有效性**——所引 `output/`／`panel/` 须可**重打开**核对数字／方向／类别；所引 DOI 须 Crossref（或等价）可解析且与出版方一致；禁止幻觉数字与空路径。**双门分工**：形式门 `/techne-dokimasia`（含 c20）与事实门 `/techne-epikrisis`（r20）分属不同单元，**不并行**（避免单轮 LLM 注意力稀释）；用户主权决定先后。操作模板、输出契约与强度分档**仅**见于 epikrisis 单元 `script/utils/techne-epikrisis/references/lens-r20.md`（部署后为 skill 根下 `references/lens-r20.md`）——本条不复述（C11·C17 协议细节单点）。

## 第五层　构造层：典则

**追溯说明**：本层是推论的几何实现；每个节点、每条规范均标注其上层出处。本层不含任何无出处的规定。

### 5.1 类型学（二维类型系统）

**类型的推导**（为何恰是这十型，不多不少）：

- 由 A1，必有"源"型：`source`；
- 由 A2，必有推导的物质载体：`script`；且派生物沿推导链展开为阶段；
- 阶段的划分由**消费者的更替**唯一决定——沿推导链，工件的消费者依次为：机器计算 → 命题断言 → 直观审阅 → 对外发表，合"给定"为五阶。任意合并两阶将丢失某推导所依赖的区分（output↔panel 合并则"机器可校验"与"人可感知"不分；panel↔figure 合并则"脚本可复现"与"声明的手工润饰"不分），故五阶不可再约：`data` → `output` → `panel` → `figure`；
- 由 A6，必有界面的物质载体：`config`；
- 由 A7，必有方法知识、结果知识与使用知识的载体：`document`（含结果报告 `reports/`）与 `README.md`；
- 由 A5，必有迹的物质载体：`HANDOFF.md`。

**类型完备性定理**：上述十型恰为公理要求的最小完备集——缺一型则某公理失去物质载体，增一型则与现有型语义重叠。证明：逐公理列其载体，构成双射（A1→source；A2→script＋阶段脊；A6→config；A7→document＋README；A5→HANDOFF）。∎

**类型总表**：

| 类型 | 维度 | 出处 | 定义与内容规范 |
|---|---|---|---|
| `source` | 阶段脊·给定 | A1 | 原始交付物（下机数据、检索导出、需求输入、依赖锁定、上游积木之果的复制件），附校验和文件；未经任何处理；不可由项目自身生成；只被上游推导读取；占用空间大者定时备份。 |
| `data` | 阶段脊·机器可读 | A2 | 链式／格式化中间数据（数据对象文件等）。三亚型：`ref/`（内化的外部参考件，自包含）；`pipeline/<dataset>/`（链式格式化产物，`{ID}_{阶段}`）；`unformatted/<dataset>/`（未格式化、未纳入链式 pipeline 的外部／第三方数据）。 |
| `output` | 阶段脊·命题断言 | A2·T3 | 面板／非可视化／前可视化分析的原始数据内容——结果表、统计表、测试报告、证据综合表；对标发表所需的 Source Data。 |
| `panel` | 阶段脊·直观审阅 | A2 | 直接从分析脚本导出的原始可视化：（1）分析、（2）质控、（3）方法／技术路线／参数的提示／引导结果。亚型：`<dataset>/`（逐数据集）与 `_summary/`（跨数据集汇总，C12 唯一例外）。 |
| `figure` | 阶段脊·对外发表 | A2·C2 | 基于面板处理后的发表级呈现；唯一允许"声明的手工润饰"的环节。 |
| `script` | 角色肋·推导 | A2 | 严格对齐 document；能力有三：（1）上游——处理 source 得链式／格式化 data；（2）下游——处理 data 得下一步 data／output／panel；（3）复现——处理 data 直接得到／复现所有 panel／output。内部三分：`pipeline/`（通用管线，可复用引擎，分上游／下游，不感知任何具体数据集）、`datasets/<dataset>/`（数据集特异应用，含统一入口）、`utils/`（共享工具函数，按功能模块化；数据集感知 IO；依赖单向，C13）。 |
| `config` | 角色肋·界面 | A6·T8 | 被脚本读取的可执行配置层（格式不拘——sh／yaml／toml／py 皆可——唯约束：双层、唯一机器点、脚本经 `Sys.getenv()/os.environ` 或等价机制读取、零硬编码绝对路径）。它是参数之**机读源**，区别于 `document/parameters/` 所装的参数*文档*。双层见 §5.4-1/2。 |
| `document` | 角色肋·方法 | A7·T12 | 六亚型：`research/`（背景调研——输入侧，形式完备、不可还原，C15）；`reports/`（结果报告——输出侧，命题—证据—结论＋解读，§5.4-13）；`methodology/`（方法学；一名一实锚点；**每个末端文件夹仅一份 md**，文首兼入口，C16·§5.3-4）；`metadata/<dataset>/`（样本／对象级元信息，内容规范 §5.4-15）；`dataset/`（数据集级登记：来源／可得性／证据，§5.4-14）；`parameters/`（参数文档，集中存档）。 |
| `README.md` | 角色肋·使用 | A7·T12 | 为使用者（人／Agent）撰写的上下文与指引（知识体）；模式见 §5.4-3。 |
| `HANDOFF.md` | 角色肋·轨迹 | A5·T12 | 为维护者（人／Agent）撰写的轨迹（历史体，不可压缩）；模式见 §5.4-4。 |

**外部依赖的归宿**［T5·C3］：一切项目外资产（数据库、远端服务、他项目产出）二选一——**内化**（复制入 `data/ref/`，附校验和）或**显式化**（登记于环境层，经界面参数只读引用）；无第三条路。

**类型互斥判定**：任一工件沿三问定位——谁生成（外部交付→source；脚本→其消费者对应的阶段型；人写→文档类）；谁是消费者（机器计算→data；命题断言→output；直观审阅→panel；对外发表→figure）；是何载体（可执行→script；被读取的机读配置→config；解释性长文→document 六亚型——前置／外部知识→research，本项目结果之综合解读→reports，样本级表格→metadata，数据集级登记→dataset；入口指引→README；决策轨迹→HANDOFF）。一物多型或无型之物，皆为违约（A4）。

**管线—变体对偶**：通用管线与数据集特异应用的分离，与配置的环境层／变体层分离是同一对偶的两面：管线是**可复用引擎**，特异应用是引擎在某数据集上的**具体执行**；环境层是**适配**，变体层是引擎在某分析方案上的**具体应用**。［A4·A6·T8·T11］

### 5.2 典则目录树（总形）

所有项目共享此树（T6）。顶层类型不可增删、不可改名（C4）；阶段脊允许空实例（如纯软件项目 `figure/` 可为空）——空阶段是"该消费形态的平凡实例"，合并是"类型的坍缩"。类型内部的子目录组织属于项目自由（附录 C7）。单源项目以单一 `<dataset>` 标识实例化（等价于该层缺省）；多源项目依 C12 强制启用 `<dataset>/` 层。

```
<项目名>/
├── README.md                          使用文档（态·知识体）[A7]
├── HANDOFF.md                         开发文档（迹·历史体）[A5]
├── config/                            界面（被脚本读取的可执行配置层）[A6]
│   ├── run_config.<ext>               环境层：唯一机器特定点（机器路径／解释器／系统工具／字体／
│   │                                  源文件名／路径派生模板／共享默认参数／外部资产登记）
│   └── <variant>.<ext>                变体层：每分析方案一件 run-spec（读取环境层后仅覆盖差异参数）
├── source/<dataset>/                  源：原始交付物，附校验和 [A1]
├── data/                              数：机器可读中间产物 [A2]
│   ├── ref/                           内化的外部参考件，自包含
│   ├── pipeline/<dataset>/            链式格式化产物（{ID}_{阶段}）
│   └── unformatted/<dataset>/         未纳入链式 pipeline 的外部／第三方数据
├── output/<dataset>/                  果：命题断言型产物，对标发表级 Source Data [A2·T3]
│   └── <variant>/logs/                结构化运行日志 [C8]
├── panel/<dataset>/                   面：脚本直接导出的原始可视化（分析／质控／方法引导）[A2]
│   └── _summary/                      跨数据集汇总面板（C12 唯一例外）
├── figure/<dataset>/                  图：润饰后的发表级呈现 [A2·C2]
├── document/                          方法文档 [A7]
│   ├── research/                      背景调研（输入侧；形式完备、不可还原，C15）
│   ├── reports/                       结果报告（输出侧；命题—证据—结论＋解读，§5.4-13）
│   ├── methodology/                   方法学（末端唯一 md·一名一实锚点＋文首入口，C16）
│   ├── metadata/<dataset>/            样本／对象级元信息（内容规范 §5.4-15）
│   ├── dataset/                       数据集级登记（来源／可得性／证据，§5.4-14）
│   └── parameters/                    参数文档（推荐值／取值范围／选择依据）
└── script/                            推导 [A2]
    ├── pipeline/                      通用管线（可复用引擎，不感知具体数据集）
    │   ├── 上游/                      source → data/pipeline
    │   └── 下游/                      data/pipeline → 下游 data／output／panel
    ├── datasets/<dataset>/            数据集特异应用（含统一入口，§5.4-9）
    └── utils/                         共享工具（数据集感知 IO；不反向依赖，C13）
```

变体输出按 `<variant>/` 子目录隔离于 output｜figure｜panel｜data/pipeline｜document/metadata 之下（C9）。

### 5.3 命名制度

1. 数据集 ID 稳定、唯一、全局一致；人类可读、机器可解析；不含空格与特殊字符；推荐形式 `<study>_<cohort>_<modality>` 或 `<study>_<id>`；同一数据集在所有 `<类>/<dataset>/`、变量、日志、文档中使用**完全相同**的字符串；同义别名禁止，确需者以显式别名表声明。［C10·C12·A4］
2. 链式中间数据：`{ID}_{阶段}`，阶段序即推导拓扑序。［C10］
3. 变体名三处同一：`config/<variant>.<ext>` ＝ 参数 `VARIANT` ＝ 输出子目录 `<variant>/`。［C9］
4. **方法学锚点 · 末端唯一 · 发表级文风**［A2·A4·A7·C11·C12·C16］：script 与 `document/methodology` 一名一实——“实”是**可链接锚点**（层内唯一正文中的二级标题，或大型项目中按 `<dataset>/` 拆叶后的唯一正文），不是文件计数。`document/` 每个最终末端文件夹（叶子）有且仅有一份 `.md`；该正文文首承担 C16 入口义务；md 不得裸露于 `document/` 根。methodology／research／reports 正文对标发表级 Methods／Results：主语明确、动词前置、术语首次定义、论断有证据；禁止无主语祈使链与无叙事弧线的条目堆叠。
5. 态内工件命名禁用时态词与版本词（v2、更新版、final、已纠正）——版本感属于迹，不属于名。［C5］
6. 目录与文件命名一律小写字母、数字、下划线、点号；命名必须反映内容。［A4·A7］

### 5.4 界面规范

**（1）环境层 `config/run_config.<ext>`**［T8］：全项目唯一的机器特定点。内容：机器路径、解释器、系统工具、字体、源文件名、**路径派生模板**（派生函数／方法，所有具体路径由此派生，变体层可覆盖以支持变体子目录）、全变体共享默认参数、**外部资产登记**（T5 显式化的只读引用）。换机／换账号／换平台只改此文件；全项目仅此一个"机器配置"文件，杜绝根目录散落多个看似机器配置的文件。

**（2）变体层 `config/<variant>.<ext>`**［T8·T11］：每个分析方案一件 run-spec；读取环境层后**仅覆盖变体专属参数**（分组／切点／阈值／VARIANT／输出子目录），绝不重复机器信息。新增分析 ＝ 新增一件 `<variant>`，不改共享文件；输出按 C9 隔离。

**（3）README 模式**［A7·T12］：文首设**入口层**（C16）——三分钟可读：一句话定位（本项目是什么、为谁而在）；读者路由（不同读者各读何节、按何顺序）；最快上手路径（一条可立即执行的最小命令／阅读线）。入口层以读者分层复述要旨并指向正节——路由而非第二居所（如论文之摘要）。随后七节齐备：

1. 项目来源、目的、第一性原理；
2. **谱系**（出生证明，见第七层）：父积木、复合方式、所依典则版本；
3. 每一个文件夹与文件的内容释义；
4. 每一个脚本／CLI／可执行内容的使用方法（含环境搭建与最常用命令）；
5. 使用 script 逐步／逐行复现 data、output、panel 的操作引导；
6. 交付物清单与下游使用指引；
7. 外部依赖清单及其自包含位置（`data/ref/`）或界面声明，及迁移注意。

**（4）HANDOFF 模式**［A5·T9·T12］：文首为**最新状态快照**（当前进度／现存问题／后续方向）；其下为**倒序日志**，新条目置于文首，每条目含：日期·会话·**决策与理由**·**参数／方法论选择的依据**·排查轨迹（含已修复项）·进度·问题·方向。只增、不删、不改——它是项目不可压缩的记忆。条目**自足可读**：每条目含背景一句（缘何而起）·决策·理由·候选·影响，单独一条亦可被未来读者无上下文理解；快照即**执行摘要**——现状／风险／方向，任何读者 60 秒可得项目全貌。**条目 6 件齐备与居所唯一见 C21；本段只定 HANDOFF 模式骨架（快照＋倒序日志＋自足可读），不复述 6 件清单。append-only（A5）。**

**（5）methodology 最小必含**［A7·T12·C16］：层内**唯一**正文（推荐 `methodology.md`）文首即入口——①阅读路径；②整体叙事（管线全景、设计理由、组件衔接，详尽直白如论文 Methods）；③节锚点索引（每脚本／组件一节，附一句职能注脚）。每节：直白摘要；职责；方法学（原理／文献／标准）；关键参数与默认值；校验手段；上下游依赖；参考文献（Crossref 核验 DOI，拆入各节）。禁止在同层并列第二份方法学 md。

**（6）parameters 最小必含**［T12］：参数名；默认值；取值范围或备选；选择依据（哪条公理／哪篇文献／哪次实测）。

**（7）research 最小必含**［C15］：命题—论证—结论的完整逻辑链；不可由项目内其他文档还原（README／HANDOFF／methodology／parameters 若不引用它即缺失关键信息，方合格）；自包含的可审计引用清单（**鼓励** DOI／PMID 等文献标识，须属本项目论证所需且可核验）；适用范围与适用条件。**闭包表述**：禁止把**他项目**的实例表／实例值／未内化路径／外部产品专名堆入本元项目态；判例可用抽象缺陷类别，或仅用本项目已登记标识（C15）。

**（8）数据集感知 IO**［C12·C13］：utils 的写出函数读取当前数据集标识（显式参数或运行时选项），写入 `<类>/<dataset>/`；禁止把数据集名硬编码进路径；保证多数据集下 source→data→output→panel 链可复现、互不串扰。

**（9）入口协议**［C14］：每个数据集特异应用暴露统一入口（形如 `run_all <variant> [steps...]`）：读取环境层 → 读取变体层 → 按拓扑序串行／并行执行各步；失败即停（`set -euo pipefail` 或等价机制）；已存在的合规产物默认跳过，显式开关（如 `FORCE=1`）强制重跑。入口脚本薄而稳定——变体参数的消费方是管线脚本，不是入口。

**（10）日志规范**［C8·T9］：每个运行步骤产生结构化日志——步骤名、起止时间、关键参数、摘要、错误位置；纯文本、可机检；存于 `output/<variant>/logs/`。

**（11）版本规范**［A5·T3］：git 是唯一的过程态载体（HANDOFF 之外）。跟踪：态文档与推导（README·HANDOFF·config·script·document）；排除：source／data／output／panel／figure 等可由链再生或体量过大的产物、临时件、IDE 元数据。提交原子化（一次提交 ＝ 一个完整意图），信息格式「动词(范围): 描述」；分支惯例属项目自由，唯要求主分支稳定。

**（12）文档三角的独立性**［A7·T12］：README（使用）、HANDOFF（轨迹）、document（方法）三面各自独立可读、独立可审计；任一面缺失，其他两面不应崩塌——但每一面都不可省略（§5.5-12）。

**（13）reports 最小必含**［A7·T12·C20·T5］：结果报告是结果性知识的唯一居所。每轮产生分析结果的工作必须出具报告（md 为默认，自包含 html 亦可），内含：命题—证据—结论完整链；**每一判断／结论句后立即挂靠证据**（C20）——项目发现挂 `output/`／`panel/` 规范路径（不复制大表）；方法／先验／外部事实挂**可解析文献标识**（DOI 等，**鼓励**，Crossref 或等价核验），并与 methodology 参考文献节互指；分析者的综合判断在文中**冻结**（C7），冻结句同样须挂靠。推荐体例：文首声明引用约定；证据用「证据｜路径或标识｜读出」表或等价紧凑列表；结论逐条绑定工件，禁止先写断言后补「见上」。**闭包（非禁文献）**：禁止把**他项目**的报告实例、未内化路径、外部 skill／agent 产品名、他仓专有过程细节当作本元项目 reports 的证据堆砌；引用外部实践时抽象为模式（路径绑定、双门分工、证据账本），或仅引用本项目已内化／已登记之文献与工件。判例：结论是「世界是什么」入 research（仍须文献可审计）；结论是「本项目发现了什么」入 reports（须 (i)，必要时加 (ii)）。

**（14）dataset 登记规范**［A1·T5·T9］：每个被引用／接续的数据集一条登记，字段齐备：dataset_id（§5.3-1）；标题／描述；来源（仓库／文献／上游积木，含 accession 或 DOI）；**可得性类别**（受控词表：`freely_downloadable`／`registration_required`／`application_required`／`on_request_only`／`internal`／`none`）；**可分析性**（受控词表：`raw_pipeline`／`processed_matrix_only`／`summary_only`／`none`）；位置（`source/<dataset>/` 或界面登记）；校验和；接入日期；许可／使用限制；**证据摘记**（可得性判定的原文或出处——判定必须可审计）。单表或每数据集一件，全项目统一其一。

**（15）metadata 内容规范**［A4·C12］：样本／对象元信息是脚本可读的表格，文法如下——UTF-8 无 BOM 的 CSV／TSV；每行一个样本／对象；列名小写 snake_case，跨数据集同名同义；首列为稳定 ID 列（唯一、非空，与链式命名 `{ID}_{阶段}` 衔接）；派生列须声明权威表示与派生式（如 `run_id = modality-sample`）；枚举列声明受控值域；缺失值统一 `NA`（禁空白与混合编码）；单元格原子化（多值用显式声明的分隔符或拆行）；语义易混列（如计数与网格面积之辨）必须定义。以上声明汇于**数据字典**（data dictionary，metadata 的同伴件）。

### 5.5 禁止事项（负空间）

每一禁令守护一条定理；违之则相应性质当场崩塌。

| # | 禁令 | 所护 | 崩塌后果 |

## 第六层　制度层：流程与 SOP

**追溯说明**：每条 SOP 是第五层构造的可执行化；步骤末注出处。SOP 不改变公理系统，只使其落地。

### SOP-0　会话启动协议（任何 Agent、任何会话、任何任务）

1. 读本典则第〇层与第五层——获得形式；［A4·A7］
2. 读目标项目 `README.md`——获得上下文、用法、谱系；［A7］
3. 读 `HANDOFF.md` 顶部（状态快照与最新条目）——获得现状与最近决策；［A5］
4. 读 `config/run_config` 与相关 `<variant>`——获得环境与方案；［A6］
5. 需要方法细节时读 `document/methodology/` 层内唯一正文（文首入口＋节锚点，与 script 同名）；交付前审计门必跑（C17）；［§5.3-4·C16·C17］
6. 列出任务所需的制品类型（§5.1），输出**对齐结论**：哪些已有造物可直接复用、什么需新建、什么需调整；**未经对齐，不动工**；［A7·T9］
7. 开工。全程遵 SOP-2，收尾遵 SOP-3，交付对照 SOP-9。

### SOP-1　新建项目

1. 实例化典则树（§5.2），顶层类型一件不缺、一名不改；［C4］
2. 写 `config/run_config`（机器信息一次写全）；［T8］
3. 落 `source/`（含校验和文件），自此不改；［A1·C8］
4. 写 `README.md` 骨架七节（谱系节注明：父积木 ＝ 无／清单；典则版本）；［§5.4-3］
5. 建 `HANDOFF.md` 首条（立项调研、初始决策与理由）；［§5.4-4］
6. 写 `.gitignore`（§5.4-11），git init 并首提交；［A5］
7. 不复制任何非自包含文件入项目。［T5］

### SOP-2　会话中作业

1. **决策即记录**：每个决策（参数取值、方法论选择、方案取舍）连同理由与候选，写入当日 HANDOFF 条目——不依赖"待会儿补"；［T9］
2. **产物即归位**：每个新工件立即落入其型之规范位置，不留临时散件；新脚本先判定归属——通用引擎入 `pipeline/`、特异应用入 `datasets/<dataset>/`、共享工具入 `utils/`；［A4·C13］
3. **文档即同步**：新脚本同步在 methodology／parameters 层内唯一正文中新增对应锚点节（一名一实），并更新文首索引与叙事；禁止另起并列 md；［§5.3-4·C16］
4. **外部依赖即声明**：进 `data/ref/` 自包含，或入 `run_config` 登记，二选一，无第三条路；［C3］
5. **非确定输入即冻结**：外部 API 响应、人工判读、随机抽样先固化为 source／data 工件附校验和，再入链；［C7］
6. **共享文件零编辑**：新增能力 ＝ 新增文件（新 variant、新 script、新 methodology），不改动共享旧文件；［T8·T11］
7. **下游产物不手改**（figure 润饰除外，且按 C2 声明并记录）；［C2］
8. **结果即报告**：每轮产生分析结果的工作，同步在 `document/reports/` 出具报告（命题—证据—结论＋解读）；**每一判断／结论后挂靠** output／panel 与／或 DOI（C20）——结论不落报告或判断无挂靠即未交付；**报告同时受 C22 事实性内容审查**——挂靠形式合规但内容不实者亦属未交付；［A7·T12·§5.4-13·C20·C22］
9. **数据即登记**：接入或引用数据集时，同步在 `document/dataset/` 完成登记（来源·可得性·证据摘记）。［A1·T5·T9·§5.4-14］

### SOP-3　会话收尾

1. 验证复现链：重跑本次触及的链，或于 HANDOFF 显式声明未验证项及原因；［C1·T3］
2. 结果报告已出具并归位于 `document/reports/`（或于 HANDOFF 声明本轮无结果性产出）；［§5.4-13］
3. 完成 HANDOFF 当日条目并更新文首状态快照；［§5.4-4］
4. 态内容当前态化：清除一切过程痕迹、版本词、会话批注；［C5］
5. git 原子提交（一次提交 ＝ 一个完整意图）；［A5］
6. 对照 SOP-9 验收；交付物即新造物。［T10］

### SOP-4　新增分析变体

1. 新增 `config/<variant>`：读取环境层，仅覆盖差异参数；［C6］
2. 新增相应 `script/datasets/<dataset>/` 应用或下游管线调用；［§5.1］
3. 在 methodology 层唯一正文新增 `<variant>` 锚点节（方法学＋参数依据＋DOI 节）；［§5.3-4·C11］
4. 输出隔离于 `<variant>/` 子目录；［C9］
5. 共享文件零编辑；［T11］
6. 更新 README 变体清单；HANDOFF 记录设计依据。［T12］

### SOP-5　新增数据集

1. 原始数据入 `source/<新数据集>/`（含校验和），自此不改；［A1］
2. 样本／对象元信息入 `document/metadata/<新数据集>/`（列名稳定、跨数据集同义）；［§5.1］
3. 建 `script/datasets/<新数据集>/` 统一入口（可由最近数据集复制并改 ID）；［§5.4-9］
4. 验证：跑入口，确认输出落于 `<类>/<新数据集>/`；［C1］
5. 更新 README 变体 × 数据集矩阵；HANDOFF 记录；［T12］
6. 禁止跨数据集共享 panel 目录（汇总面板入 `panel/_summary/`）。［C12］

### SOP-7　审计（三径 × 七查）

**三径**（T9）：

1. **正向**：抽取任一派生物，由其声明的链重算并比对；［T3］
2. **逆向**：沿 `{ID}_{阶段}` 与 script 溯至 source，核对校验和；［T2·C8］
3. **叙事**：查 HANDOFF 与 git，确认每步决策有理由、每个变更有记录。［A5］

**七查**：

1. 结构查：典则树逐项在位（§5.2）；
2. 路径查：静态扫描脚本与文档，无硬编码绝对路径（§5.4-1 之外）；
3. 依赖查：无项目外隐式引用；无反向依赖（C13）；
4. 执行查：干净环境（新 clone）中端到端重跑，比对预期产物；
5. 文档查：文档三角各面满足最小必含（§5.4-3~7、13~15）；methodology 与 script 一名一实；reports 每一判断／结论满足 C20 证据挂靠（output／panel 与／或 DOI）；**事实复核 (r20) 跑过（完整性 + 事实有效性），零 violation**；dataset 登记含可得性证据；各知识层入口在位且自足（README 入口层三分钟可读、methodology 层唯一正文文首、HANDOFF 快照即执行摘要，C16）；
6. 产物查：每种 panel／output 存在 README 显式给出的复现命令；日志齐备（§5.4-10）；
7. **迹查**（c21）：HANDOFF 条目满足 6 件齐备（提示词原文+轨迹+决策+讨论+改动+见解）；6 件居所唯一（无末尾附录式汇总）。

三径皆通、七查皆过，审计通过；任一断裂即定位为缺陷并修复（修复本身亦入 HANDOFF），输出审计报告，不通过项必须有改进计划。［T9·C8］

### SOP-8　迁移（换机／换账号／换平台）

1. 仅改 `config/run_config`；［T8］
2. 全链重跑；［T3·T4］
3. 与既有产物比对：差异为零（或在 methodology 声明的容差内），迁移完成。

## 第七层　谱系层

**追溯说明**：本层由 T10（积木定理）与 A5（态迹）直接生成，规定项目之间与项目之内的时间结构。

1. **项目 DAG**：积木为节点，"接续"为有向边；因复合行为有先后，图必无环。
2. **出生证明**＝README 谱系节：父造物清单、复合方式（以 T10 推论的开放自然语言表达）、所依典则版本、创建日期。无父造物者声明"源生项目"。
3. **项目无版本号**：态永远当前（T7）；演化史在 HANDOFF 与 git。唯**典则**有版本号；项目在谱系节声明所依版本，不自动跟随新版（附录 C）。
4. **块内微谱系**：`{ID}_{阶段}` 记录工件的构造坐标；`<variant>/` 记录方案的分支坐标；二者合起来，任何工件可在推导 DAG 上被唯一指认。［T2］
5. **项目形态类型**：**原子项目**（不可再分的最小积木）、**复合项目**（多积木并列或层次组合而成）、**嵌套项目**（子目录自身即完整积木，独立满足 A1–A7）。形态差异不改变典则：任何层级都是同一棵树，复合与嵌套的封闭性由 T10 的递归保证。

### A.2 候选"公理"的还原（不可还原性的直接证据）

凡工程实践中被提议为"公理"者，在本系统中皆可证明为定理——这直接表明本七公理系统在逻辑上不可再约：

| 候选"公理" | 本系统归宿 | 还原要点 |
|---|---|---|
| 存在性（项目可独立寻址） | A4＋A6 的推论 | 寻址稳定性来自类型定址（`loc(τ)`）与界面单点，无须独立假设。 |
| 同一性（同内容同结构即同一项目） | A3 的换位 | 确定性映射的逆读：输出差异必源于输入差异。 |
| 组合性（项目组合仍是项目） | **T10** | 复合封闭性被证明，非被假设。 |
| 可追溯性（状态可溯至生成过程） | **T9** | 三径可溯是分层、复现、态迹三分的合取。 |
| 无损性（信息不得丢失） | A5 | "迹只增不删"已涵盖；且 T3 表明派生物本就可再生，无须囤积。 |
| 可复现公理 | **T3** | 复现是构造＋确定＋界面的推论。 |
| 可分解公理 | A2＋T2 | 构造性已含无环与良基；原子分解是 DAG 分层。 |
| 可定位公理 | A4（＋A3） | 类型定址＋命名唯一，是形式秩序的分内之事。 |
| 可审计公理 | **T9** | 合规可判性是三径独立的推论。 |
| 可自包含公理 | **T5** | 闭包是构造与界面两公理的合力。 |

**结论**：诸候选"公理"的每一条均可在本系统内证明；而本系统七条公理的独立性由 §2.8 的反模型总证保证。故本系统是诸候选公理化中逻辑上不可还原的最小核。∎

## 附录 B　术语表

- **积木（techne）**：满足 A1–A7、可被复合且复合产物仍合规的项目（D10，T10）。
- **态／迹（state／trace）**：项目的当前状态与变更记录，二者不交（A5）。
- **界面（interface）**：项目与环境之间唯一声明的交换点（A6）；分环境层与变体层（T8）。
- **源（source）**：项目内不可推导、不可变更的给定工件集（A1）。
- **推导（derivation）**：可执行的、输入先于输出的构造性变换（A2）。
- **管线／变体（pipeline／variant）**：通用引擎与其在特定数据集／方案上的具体应用（§5.1 对偶）。
- **谱系（genealogy）**：项目之间（父积木）与项目之内（`{ID}_{阶段}`、`<variant>/`）的时间坐标系（第七层）。
- **复合代数**：复合封闭性的开放自然语言表达（T10 直接推论，非新增规则）；禁止路径引用，禁止修改上游项目。
- **冻结（freeze）**：将非确定输入固化为附校验和的 source／data 工件（C7）。
- **读者三角**：构建者／审计者／使用者；人与 Agent 同属读者（D6）。
- **文档三角**：README（使用）·HANDOFF（轨迹）·document（方法），三面独立可读（§5.4-12）。
- **结果性知识／结果报告（reports）**：项目"发现了什么"的综合与解读；唯一居所为 `document/reports/`（T12·§5.4-13）。
- **数据集登记（dataset registry）**：数据集级的来源、可得性与证据的结构化记录；居所为 `document/dataset/`（§5.4-14）。
- **数据字典（data dictionary）**：列语义与受控词表的声明件，metadata 的同伴（§5.4-15）。
- **负空间**：§5.5 禁令集——每条禁令守护一条定理。
- **统领文档／入口层（governing synthesis／entry layer）**：每一知识居所的唯一入口——读者路由、整体叙事与域内索引（C16）；项目入口层居 README 文首，方法学入口为层内唯一正文文首（`document/methodology/methodology.md`），轨迹入口为 HANDOFF 快照。
- **末端一文件（terminal single file）**：document 每个叶子文件夹仅一份 md；一名一实以节锚点实现（§5.3-4）。
- **典则（canon）**：本文件自身；一件自指的造物（附录 C）。

## 附录 C　典则的自指、版本与修正

1. **自指一致**：本典则自身是一件造物——其知识记录于自身（A7），其修订史归于典则项目的 HANDOFF 与 git（A5），其结构遵循典则（§5.2）。凡本典则要求的，本典则自身先行满足。
2. **可被自身审计**：本文每一条公理、定理、推论、构造、SOP 均可被 SOP-7 审计。
3. **可被自身重构**：任何一条均可在不违反公理的前提下被替换为更优表述；替换后须重新执行 SOP-7 验证全文自洽。重构的目标态是**当前闭合的最小表述**，不是在旧表述旁累加例外与补丁编号。
4. **五分钟对齐**：本文须使具备通用能力的读者（人或 Agent）在五分钟内完成首次理解并开始工作——故短而密：不在显而易见的实现细节上冗述，不在罕见场景上铺张，同一规则只陈述一次。
5. **版本语义**：主版本 ＝ 公理层变更（第二层）；次版本 ＝ 定理／推论层变更（第三、四层）；修订号 ＝ 构造／制度层变更（第五、六层）。
6. **修正程序**：任何修改提案按**一次推导**处理——理由入 HANDOFF，变更入新版本；项目按 README 谱系节声明的典则版本遵循，不自动跟随新版；升级典则版本本身是该项目的一次迭代，记入其 HANDOFF。一次推导的输出必须是**再生后的当前态全文**（含 daemon 投影），不得留下“旧句＋补丁句”的双层并存。
7. **演替边界**：顶层类型的增删属于主版本事件；类型内部子目录的组织属于项目自由。
